# 最小可行配置清单

## 先说结论

如果你只是想把这条视频流水线先跑通，不需要一开始就把所有东西配满。

最小可行版本，只需要准备 5 类东西：

1. 运行环境
2. 语音合成能力
3. 时间戳提取能力
4. 图片来源
5. 一个能接收场景数据的渲染模板

---

## 一、运行环境

最基础的环境建议是：

- Node.js 18+
- Python 3.10+
- FFmpeg
- Whisper 运行环境
- 一个可调用大模型的命令行或本地执行入口

这部分的判断标准很简单：

- 能跑脚本
- 能生成音频
- 能提取时间戳
- 能把结果写进本地目录

只要这四件事成立，链路就能往下走。

---

## 二、必须配置的账号和能力

### 1. TTS 账号

用途：

- 把口播脚本变成旁白音频

至少要有：

- API Key
- Group ID 或项目标识
- 一个可用 voice_id

如果你还没克隆自己的声音，也可以先用默认音色测试链路。

### 2. 图片来源

用途：

- 给标题卡和内容卡准备背景图

最低要求：

- 一个可用的图片 API Key

如果你暂时没有图片能力，也不是不能跑，只是最后交给模板的素材会不完整。

### 3. 时间戳提取能力

用途：

- 让字幕和语音对齐

最低要求：

- 本地能运行 Whisper

这部分通常不需要额外付费，但要保证机器上能跑起来。

---

## 三、可选但强烈建议配置的内容

### 1. 品牌 CTA

建议提前准备一条统一的结尾文案，例如：

- 关注类 CTA
- 收藏类 CTA
- 私域引导 CTA

这样每条视频不会出现结尾风格飘忽的问题。

### 2. 内容素材库

如果你有自己的素材库，建议至少准备：

- 常用标题角度
- 常用钩子
- 常用金句
- 常用结构模板

没有素材库也能跑，但内容风格会更泛。

### 3. 输出目录规范

建议一开始就把输出目录分清楚：

- `output/`：最终产物和归档
- `tmp/`：临时文件

这样后面排错和复盘都会轻松很多。

---

## 四、最小可行的目录结构

建议至少有下面这几个目录：

```text
video-workflow/
├── .env
├── config/
├── scripts/
├── output/
└── tmp/
```

如果你想长期跑，再补：

```text
├── records/
├── reviews/
└── assets/
```

---

## 五、最小可行的配置项

至少要填的配置可以压缩成下面这些：

### 必填

- `MINIMAX_API_KEY`
- `MINIMAX_GROUP_ID`
- `MINIMAX_VOICE_ID`
- `PEXELS_API_KEY`
- `WORKFLOW_RECORDS_DIR`

### 建议填

- `MINIMAX_VOICE_SPEED`
- `WORKFLOW_BRAND_CTA`
- `OUTPUT_DIR`
- `TEMP_DIR`
- `SCENES_OUTPUT_PATH`

### 已有自定义模板时再填

- `RENDER_COMMAND_TEMPLATE`

---

## 六、最小可行的运行方式

第一次测试，不要直接追求全自动。

建议按这个顺序跑：

1. 先生成文案策划稿
2. 再生成口播脚本
3. 再跑语音
4. 再跑时间戳
5. 再生成场景数据
6. 最后再接模板渲染

这样一旦哪一步出问题，你能立刻定位。

---

## 七、第一次跑通时最常见的 5 个坑

### 1. 脚本能生成，但 TTS 失败

通常是：

- API Key 错
- voice_id 不可用
- 账号权限没开

### 2. 音频生成了，但时间戳很乱

通常是：

- Whisper 模型没装好
- 语言参数不对
- 音频语速过快

### 3. 场景数据生成了，但画面很怪

通常不是脚本错，而是模板接收字段和场景数据结构不一致。

### 4. 图片生成出来了，但语义不对

这很常见，所以图片节点最好人工确认。

### 5. 最终无法渲染

常见原因：

- 模板没有接 `scenes-data`
- 模板没接音频文件
- 路径引用错误

---

## 八、什么叫“已经跑通”

不是看到一两个中间文件就算跑通。

真正的“最小可行跑通”标准是：

- 有脚本
- 有音频
- 有时间戳
- 有 `scenes-data`
- 有标题图和至少一张分镜图
- 模板能消费这些文件并输出成片

到这一步，你就已经拥有了一条可迭代的视频生产链路。

---

## 九、推荐的落地顺序

如果你是第一次搭，不要从“自动发布”开始。

正确顺序是：

1. 先跑通生成链路
2. 再接模板渲染
3. 再加审核节点
4. 最后再加自动发布

因为真正决定你能不能长期跑的，不是发布脚本，而是前面的内容和素材链路是否稳定。
